## 1. Cosa esiste già (stato di maturità)

Il backend di build è sorprendentemente avanzato: non si parte da zero, si parte da un motore di scelte generico (`buildChoiceGroup`) già funzionante per razza/classe/background, più servizi dedicati per ogni dominio.

Endpoint REST attuali (`src/routes/api.mjs`):

| Metodo | Path | Funzione |
|---|---|---|
| GET | `/api/builds/draft` | Manifest di scelte disponibili (razza/classe/background/incantesimi/ability score) |
| POST | `/api/builds/validate` | Valida una selezione di scelte, ritorna errori/warning |
| POST | `/api/builds/sheet` | Costruisce la scheda PG completa risolta |
| GET | `/api/races/:slug/builder` | Dettaglio builder per una razza |
| GET | `/api/classes/:slug/builder` | Dettaglio builder per una classe |
| GET | `/api/backgrounds/:slug/builder` | Dettaglio builder per un background |
| GET | `/api/classes/:slug` | Aggregazione progressione classe (legge `raw5e.classFeatures`) |
| GET | `/api/filters/:entityType` | Metadata filtri (parziale, vedi §4.9) |

## 3. Regole da non violare (riassunto da ARCHITECTURE.md / DATABASE.md)

Da rispettare in ogni nuovo lavoro sul builder, per non introdurre regressioni nel resto della piattaforma:

1. **`_id` > slug** — gli slug non sono univoci globalmente (es. `spellcasting`, `goblin`). Ogni nuova logica deve referenziare entità per `_id` completo (`entityType:source:slug`).
2. **`content.en` è canonico** — mai scrivere logica di gameplay che dipende da `content.it` (le traduzioni possono essere rigenerate/azzerate).
3. **`deprecated: true` va sempre escluso** dalle query usate dal builder (alcune `classFeature`/`subclassFeature` legacy sono state sostituite da ID class-aware).
4. **`raw5e` non si tocca** — è la fonte originale 5etools, serve da fallback quando `normalized` è incompleto (frequente per `classFeature`).
5. **Mai eseguire `scripts/import-mongo.mjs` senza dry-run** — cancella e reinserisce i documenti edizione 2014.
6. Prima di modifiche ampie sul DB: `mongodump` o export dei record coinvolti (vedi §19 di `DATABASE.md`).

#### Milestone 4c — Multiclasse

**Analisi fatta il 2026-06-25, prima di scrivere codice (su richiesta esplicita).** Spezzata a sua volta in 3 passi indipendentemente testabili, in ordine di rischio crescente.

##### Dati già disponibili (niente da inventare a memoria)

Verificato in MongoDB: ogni documento `class` ha già `raw5e.multiclassing` con **prerequisiti** e **competenze guadagnate multiclasse** (diverse da quelle di partenza al 1° livello), importati da 5etools — non serve hardcodare nulla:
```
fighter.multiclassing = { requirements: {or:[{str:13,dex:13}]}, proficienciesGained: {armor:[light,medium,shield], weapons:[simple,martial]} }
monk.multiclassing    = { requirements: {dex:13, wis:13} }  // richiede ENTRAMBE, non "or"
sorcerer/wizard.multiclassing = { requirements: {cha:13} } / { requirements: {int:13} }  // senza proficienciesGained: nessuna competenza extra
```
Verificato che ogni classe usa una chiave diversa (`or` per "una delle due", più chiavi dirette per "entrambe necessarie") — la validazione deve gestire entrambe le forme.

La **tabella slot incantesimo multiclasse** non va hardcodata: è identica, per regola RAW, alla tabella di un qualsiasi incantatore completo a slot pieni (es. `wizard.normalized.spellcasting.spellSlotsProgression`) — basta riusarla indicizzandola per il "livello incantatore multiclasse combinato".

**Contributo al livello incantatore combinato per classe** (verificato dove possibile nel dataset, una voce confermata via ricerca web perché incerta a memoria):
- Incantatori completi (bardo, chierico, druido, stregone, mago — `casterProgression:"full"`): contano il **livello pieno**
- Mezzi incantatori (paladino, ranger — `casterProgression:"1/2"`): **livello/2 arrotondato in giù**
- Artefice (`casterProgression:"artificer"`): **livello/2 arrotondato in su** — eccezione unica confermata via ricerca (Tasha's Cauldron), diversa dai mezzi incantatori normali
- Warlock (Magia del Patto, `casterProgression:"pact"`): **non contribuisce mai** al pool condiviso, resta sempre separato anche in multiclasse
- Terzi incantatori da sottoclasse (Cavaliere Arcano guerriero, Ladro Arcano furfante): **fuori scope** — questo dataset non taggia `casterProgression` sulle sottoclassi, andrebbe hardcodato un caso speciale; rimando esplicito, non li tratto come incantatori multiclasse per ora (non rompono nulla, semplicemente non contribuiscono al pool, comportamento già corretto per chi non ha quella sottoclasse)

##### Cosa cambia per davvero (mappatura del codice)

Il livello va diviso in due concetti distinti, oggi confusi in uno solo:
- **`characterLevel`** (somma di tutti i livelli di classe): guida bonus competenza, conteggio totale dadi vita, lookup nella tabella slot multiclasse
- **`classLevel[slug]`** (livello in *quella* classe specifica): guida tutto il resto — ASI (ai livelli 4/8/12/16/19 *di quella classe*, non del personaggio), progressione incantesimi di quella classe, scelte di sottoclasse, dadi specifici classe (ira/ki/punti stregoneria/Channel Divinity dalla Milestone 4a, che oggi leggono `sheet.identity.level` cioè il livello totale — **da correggere** quando si introduce `classLevel`)

Buona notizia verificata leggendo il codice: le funzioni "per classe" già esistenti (`buildClassSnapshot`, `buildClassBuilderSpellcastingData`, `buildClassBuilderFeatureData`, `buildClassBuilderLoadoutData`, `buildClassOptionalFeatureData`) accettano già un parametro `level` indipendente — oggi viene chiamata *una sola volta* passando il livello totale perché c'è una sola classe. Per il multiclasse basta chiamarle **una volta per classe**, passando il livello di *quella* classe — non serve riscriverle.

##### Schema proposto (retrocompatibile)

- Richiesta: resta valido `class`/`classSource`/`level`/`subclass` (singolo, invariato — i 195 test esistenti non si toccano). Nuovo, alternativo: `classes: [{slug, source, level, subclass}, ...]` (per `GET /builds/draft` come query JSON-encoded, per le POST come array nel body). Il primo elemento è sempre "la classe di partenza" (unica a ricevere equipaggiamento iniziale).
- ID delle scelte: invariati quando c'è una sola classe (`class.subclass`, `class.skills.0`, `class.spells.cantrips`, `class.abilityScoreImprovement.4`). Con 2+ classi, prefissati con lo slug per disambiguare: `class.fighter.subclass`, `class.wizard.spells.cantrips`, `class.fighter.abilityScoreImprovement.4`. Le competenze guadagnate da multiclasse (dalla 2ª classe in poi) diventano scelte/grant a parte, non equipaggiamento.
- Risposta: nuovo `draft.classes`/`sheet.identity.classes` (array, sempre presente); `draft.class`/`sheet.identity.class`/`.subclass` restano come alias del primo elemento per chi consuma l'API oggi.
